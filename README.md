# DongSeoulUniversity_python_26.4.16_7week_tetris

응. 이전 HTML 계획서 내용을 기준으로, 지금 tetris_menu_updated.py에 어떤 요소가 실제 코드로 들어갔는지 코드 줄번호 기준으로 정리해줄게.

먼저 핵심부터 말하면, 이 코드는 HTML 계획서의 기본 테트리스 뼈대를 가져오고, 그 위에 네가 나중에 요청한 메뉴 / 난이도 / 3개 미리보기 / 마우스 조작 / 도움말 스크롤 같은 확장 기능을 덧붙인 구조야.
즉, HTML 내용 100% 그대로 복붙된 게 아니라, 계획서를 코드 구조로 변형해서 반영한 것이야.
계획서의 핵심 축은 "블록, 격자판, 충돌, 낙하, 줄 삭제, 점수, 게임 루프, 다음 블록"이었고, 그 부분은 실제 코드에 들어가 있어.

1. HTML의 "게임 정의 / 테트리스 기본 개념"
   - HTML에서 말한
     "위에서 떨어지는 블록을 조종해서 빈틈 없이 쌓고, 가득 찬 줄을 없애 점수를 올리는 게임"
     이 핵심 정의는 코드 전체 구조로 반영됐어.

   - 코드 위치
     - 47~213줄 : 7종 블록 모양과 회전 상태 정의
     - 252~460줄 : 게임 핵심 로직 클래스 TetrisGame
     - 380~386줄 : 줄 삭제
     - 365~375줄 : 점수와 레벨 계산

   - 역할
     - HTML의 "테트리스가 어떤 게임인가" 설명이 코드에서는
       - 블록 데이터
       - 블록 이동 / 회전
       - 줄 삭제
       - 점수 증가
       이 4개 축으로 실제 구현된 거야.

2. HTML의 "색상 팔레트 / 이미지 없이 rect로 그리기"
   - HTML에서 이미지 없이 pygame.draw.rect()로 블록을 직접 그리고, 색상 팔레트를 미리 정해둔다고 했던 부분이 그대로 들어가 있어.

   - 코드 위치
     - 22~42줄 : 배경색, 패널색, 글자색, 블록 색상 정의
     - 495~506줄 : 다음 블록 미리보기 박스 그리기
     - 509~557줄 : 보드 칸, 놓인 블록, 현재 블록 그리기

   - 역할
     - COLOR_MAP 이 HTML의 색상표 역할
     - draw_board() 가 HTML의 "rect로 직접 그리기" 역할
     - draw_piece_mini() 가 다음 블록 미리보기 역할

3. HTML의 "화면 스케치"
   - HTML에는 왼쪽 격자판, 오른쪽 점수 / 다음 블록 패널이 있는 화면 스케치가 있었지.
     그 배치가 코드에 반영돼 있어.

   - 코드 위치
     - 10~19줄 : 화면 크기, 칸 크기, 보드 위치, 오른쪽 패널 위치
     - 509~557줄 : 왼쪽 플레이 보드
     - 560~588줄 : 오른쪽 정보 패널

   - 역할
     - PLAY_X, PLAY_Y, PLAY_W, PLAY_H 는 왼쪽 게임판 위치
     - SIDE_X 는 오른쪽 UI 패널 시작점
     - draw_side_panel() 에서 점수, 라인, 레벨, 난이도, 다음 블록 3개, 조작 설명을 그림

   - 정리
     - HTML의 스케치가 완전히 똑같은 픽셀 값으로 간 건 아니지만,
       "왼쪽 게임판 + 오른쪽 정보창" 구조는 그대로 반영됐어.

4. HTML의 "7종 블록 + 회전 상태"
   - HTML에서 BLOCK_DATA 처럼 7종류 블록 × 회전 상태를 미리 정의한다고 했던 부분이
     실제 코드에서는 문자열 4x4 방식으로 들어갔어.

   - 코드 위치
     - 47~213줄 : PIECES

   - 역할
     - "I", "O", "T", "S", "Z", "J", "L" 각 블록 정의
     - 각 블록마다 회전 상태 4개 준비
     - HTML의 1차원 튜플 방식 대신, 여기서는
       "....", ".TT.", ...
       같은 문자 배열 방식으로 바뀜

   - 정리
     - HTML : BLOCK_DATA
     - 코드 : PIECES
     - 표현 방식만 달라졌고 의미는 동일해.

5. HTML의 "현재 블록 / 다음 블록 / 게임 상태 변수"
   - HTML에서 전역 변수나 Block 객체로 설명했던
     "현재 블록, 다음 블록, 점수, 라인, 게임오버 상태"가
     코드에서는 TetrisGame 클래스 안으로 들어갔어.

   - 코드 위치
     - 252~259줄 : 초기 설정값
     - 273~286줄 : reset() 에서 게임 상태 초기화
     - 298~308줄 : 새 블록 생성
     - 279~285줄 : 점수, 라인, 레벨, 게임오버, 타이머

   - 역할
     - self.board  -> HTML의 FIELD 역할
     - self.current -> 현재 낙하 블록
     - self.queue -> 다음 블록 대기열
     - self.score, self.lines, self.level, self.game_over -> 게임 상태

   - 정리
     - HTML에서는 설명용으로 전역 변수 중심이었고,
       실제 코드는 객체지향으로 묶어서 관리한 거야.

6. HTML의 "게임 시작 시 랜덤 블록 생성"
   - HTML에서 "게임 시작하면 랜덤 블록이 맨 위에서 나타난다"는 설명이 있었는데,
     그건 다음 부분이 담당해.

   - 코드 위치
     - 288~296줄 : 블록 랜덤 섞기
     - 298~308줄 : spawn_new_piece()

   - 역할
     - 7-bag 방식으로 블록을 섞어 큐에 넣음
     - 새 블록을 x=3, y=-1 위치에 생성
     - 생성하자마자 충돌하면 게임오버 처리

   - 정리
     - HTML은 "랜덤 블록 등장" 수준으로 설명했고,
       실제 코드는 그걸 bag 시스템으로 좀 더 안정적으로 구현했어.

7. HTML의 "좌우 이동 / 빠른 낙하 / 회전 / 충돌 검사"
   - HTML에서
     - 방향키로 이동
     - 충돌 없으면 적용
     - 회전
     - 아래로 빠르게 내리기
     를 설명했는데, 이 부분이 가장 직접적으로 들어간 구간이야.

   - 코드 위치
     - 310~319줄 : collision() 충돌 판정
     - 324~335줄 : move() 이동
     - 337~352줄 : rotate() 회전
     - 400~411줄 : soft_drop_step() 아래로 빠르게 내리기
     - 413~420줄 : hard_drop() 하드 드롭
     - 994~1011줄 : 실제 키 입력 처리

   - 역할
     - collision() = HTML의 is_overlapped() 역할
     - move() = 좌우 / 아래 이동
     - rotate() = 회전
     - soft_drop_step() = ↓
     - hard_drop() = SPACE

   - 차이점
     - HTML은 SPACE = 회전 예시였는데,
       네 추가 요청 때문에 실제 코드에서는
       - R = 시계 회전
       - Q = 반시계 회전
       - SPACE = 하드 드롭
       으로 바뀌었어.

8. HTML의 "줄 소거"
   - HTML에서 erase_line() 으로 설명한 부분은 실제로 clear_lines() 로 구현됐어.

   - 코드 위치
     - 380~386줄

   - 역할
     - 빈칸이 있는 줄만 남기고
     - 가득 찬 줄은 제거
     - 위에 빈 줄을 다시 추가

   - 차이점
     - HTML은 del FIELD[ypos] 방식이었고,
       이 코드는 리스트 재구성 방식이야.

   - 정리
     - 방법은 다르지만 결과는 같아.
       - 꽉 찬 줄 제거
       - 위 블록 내려옴

9. HTML의 "블록 고정"
   - HTML에서 "블록이 더 내려갈 수 없으면 FIELD에 기록"이라고 했던 부분은 여기야.

   - 코드 위치
     - 354~378줄 : lock_piece()

   - 역할
     - 현재 블록 칸들을 보드에 확정 저장
     - 줄 삭제 실행
     - 점수 반영
     - 라인 수 반영
     - 레벨 갱신
     - 다음 블록 생성

   - 정리
     - 이 함수가 HTML 설명의
       - 블록 착지
       - 보드에 고정
       - 줄 삭제
       - 다음 블록 등장
       을 한 번에 처리하는 핵심 함수야.

10. HTML의 "점수 계산"
    - HTML에서는 (2 ** erased) * 100 처럼 설명했는데,
      실제 코드는 일반적인 테트리스 점수 방식에 더 가까운 형태로 바뀌었어.

    - 코드 위치
      - 365~372줄

    - 역할
      - 1줄 : 100 * (level + 1)
      - 2줄 : 300 * (level + 1)
      - 3줄 : 500 * (level + 1)
      - 4줄 이상 : 800 * (level + 1)

    - 정리
      - HTML의 "줄을 많이 지울수록 점수 더 많이 받음" 개념은 유지했고,
        공식만 좀 더 테트리스답게 수정한 거야.

11. HTML의 "레벨 증가 / 속도 증가"
    - HTML에서 "시간이 지나거나 진행될수록 블록이 빨라진다",
      "난이도 증가"를 설명했는데,
      이건 실제 코드에서 꽤 강하게 반영됐어.

    - 코드 위치
      - 215~218줄 : 레벨별 낙하 속도표 LEVEL_FRAMES
      - 242~249줄 : 레벨에 따른 중력 시간 계산
      - 374~375줄 : 라인 수 기준 레벨 상승
      - 422~454줄 : 자동 낙하 속도 적용

    - 역할
      - 10줄 지울 때마다 레벨 1 증가
      - 레벨이 오를수록 자동 낙하 간격 감소
      - 네 요청대로 "일반적인 테트리스 레벨 상승 방식" 쪽으로 구현됨

    - 정리
      - 이 부분은 HTML의 "INTERVAL 감소" 설명을
        실전형 테트리스 속도표로 바꾼 것이라고 보면 돼.

12. HTML의 "다음 블록 미리보기"
    - HTML에서 다음 블록 표시가 있었고,
      코드에서는 더 확장해서 3개 미리보기로 넣었어.

    - 코드 위치
      - 456~459줄 : preview_names(3)
      - 570~572줄 : 오른쪽 패널에 3개 표시
      - 495~506줄 : 미니 블록 그림

    - 정리
      - HTML의 STEP 7 "다음 블록 보이기"가 실제 코드에서는
        강화된 버전으로 구현된 거야.

13. HTML의 "게임 오버"
    - HTML에서 "맨 위까지 쌓이면 GAME OVER"라고 했던 부분은 여기서 반영돼 있어.

    - 코드 위치
      - 307~308줄 : 새 블록 생성 시 충돌하면 게임오버
      - 590~596줄 : 게임오버 화면 표시
      - 1048~1049줄 : 실제 렌더링

    - 주의할 점
      - HTML은 단순 설명이었고,
      - 실제 코드는 "새 블록이 생성될 자리가 막혀 있으면 게임오버" 방식으로 처리해.
      - 이게 보통 더 정확한 구현이야.

14. HTML의 "게임 루프"
    - HTML의 가장 중요한 부분 중 하나였던
      "이벤트 확인 → 데이터 업데이트 → 화면 그리기"
      흐름이 실제 코드의 main() 에 그대로 들어가 있어.

    - 코드 위치
      - 928~1060줄 전체
      - 특히
        - 950~1019줄 : 이벤트 처리
        - 1021~1042줄 : 게임 상태 업데이트
        - 1044~1054줄 : 화면 그리기

    - 역할
      - HTML의 게임 루프 설명을 실제 실행 루프로 구현
      - 닫기 이벤트, 키 입력, 마우스 입력
      - 반복 이동
      - 자동 낙하
      - 보드 그리기
      - 메뉴 / 게임오버 오버레이 표시

    - 정리
      - 이 구간은 HTML의 "게임이 돌아가는 순서"를 가장 직접적으로 코드로 옮긴 부분이야.

16. HTML에는 없었고, 이번 코드에서 추가된 것
    - 이건 반대로 계획서에는 없는데,

- HTML 계획서의 핵심 반영
  - 블록 7종과 회전 : 47~213줄

- HTML 이후 추가 확장
  - 난이도 쉬움 / 보통 / 어려움 : 44~46줄, 265~271줄, 388~455줄

정리하면:
- HTML 계획서의 핵심 테트리스 구조는 실제 코드에 대부분 들어가 있다.
- 다만 계획서 그대로 1:1 복붙된 것이 아니라, 객체 구조와 실제 플레이 방식에 맞게 변형되어 구현되었다.
- 그리고 메뉴, 난이도, 도움말, 스크롤, 마우스 조작, 고스트 블록 같은 기능은 계획서 이후 추가 확장된 부분이다.
